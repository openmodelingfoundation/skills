# Analysis Skills Suite — Product Specification

**Version:** 0.2-draft
**Status:** Handoff — ready for implementation planning
**Audience:** Engineering lead, contributing developers
**Target Skills Audience:** Researchers who code, research software engineers

---

## 1. Purpose

Define a modular ecosystem of reusable agent skills for reproducible scientific data analysis and computational modeling workflows.

The system supports:

- computational social science
- agent-based and simulation output analysis
- geospatial workflows
- NLP and text analysis
- statistical analysis and uncertainty quantification

Primary goals: reproducibility, modular orchestration, methodological transparency, open-science interoperability, agent composability.

---

## 2. Design Principles

These principles govern all implementation decisions. When they conflict, earlier ones take precedence.

1. **Reproducibility-first** — every execution must be fully re-runnable from its provenance record
2. **Human-in-the-loop** — routing decisions and consequential choices require user confirmation before execution
3. **Provenance-aware** — all inputs, parameters, intermediate outputs, and software versions are recorded in open, analysis-friendly formats
4. **Uncertainty-explicit** — outputs include uncertainty estimates wherever the method supports them
5. **Modular composition** — skills are independently testable and composable without circular dependencies
6. **Transparent methodology** — analytical choices are clear to the user, not buried in defaults
7. **Open-source preferred** — proprietary dependencies require explicit justification
8. **FAIR-compatible** — outputs conform to Findable, Accessible, Interoperable, Reusable principles and adhere to associated Software and Data Management Plans if available

### Non-Goals

- Fully autonomous scientific reasoning
- Replacing domain expertise
- Black-box analysis pipelines
- Proprietary platform lock-in

## 3. System Architecture

### 3.1 Component Overview

```text
User Request
     │
     ▼
┌─────────────────────────────────────┐
│           Orchestrator              │
│  intent → routing → DAG → dispatch  │
└──────────────┬──────────────────────┘
               │  SkillRequest
               ▼
┌──────────────────────────────────────────────────────┐
│                   Skill Registry                     │
│                                                      │
│  statistical  │  sensitivity  │  geospatial          │
│  text         │  network      │  visualization       │
└──────────────────────────────────────────────────────┘
               │  SkillResult
               ▼
┌─────────────────────────────────────┐
│         Provenance Aggregator       │
│         assembles RO-Crate bundle   │
└──────────────┬──────────────────────┘
               ▼
          Final Output
```

### 3.2 Skill Independence Constraint

Skills may not call other skills directly. All inter-skill coordination is the orchestrator's responsibility. This constraint must be enforced by an integration test (see §10.3). Skill executions MUST be isolated from one another.

## 4. Interface Contracts

### 4.1 SkillRequest (Orchestrator → Skill)

```python
@dataclass
class SkillRequest:
    skill_id: str                        # e.g. "sensitivity-analysis"
    task_id: str                         # UUID, generated by orchestrator
    inputs: dict[str, DataRef]           # named references to input data
    parameters: dict[str, Any]           # skill-specific config
    environment: EnvironmentSpec         # conda/uv/pip-tools spec
    provenance_context: ProvenanceRecord # upstream lineage to carry forward
    human_confirmation_required: bool    # True for consequential parameter choices
    schema_version: str                  # SemVer / calver for this dataclass schema, should be compatible with SkillResult
```

### 4.2 SkillResult (Skill → Orchestrator)

```python
@dataclass
class SkillResult:
    task_id: str
    skill_id: str
    status: Literal["success", "partial", "failed"]
    outputs: dict[str, DataRef]          # named output artifacts
    warnings: list[Warning]              # non-fatal issues (see §4.4)
    errors: list[Error]                  # fatal issues
    provenance: ProvenanceRecord         # inputs + software + parameters used
    diagnostics: DiagnosticBundle        # residuals, convergence, assumption checks
    methods_summary: str                 # human-readable description of what ran
    schema_version: str                  # SemVer / calver for this dataclass schema, should be compatible with SkillRequest
```

### 4.3 DataRef

Persisted artifacts are immutable once registered in provenance.

```python
@dataclass
class DataRef:
    uri: str                             # file path or content-addressable hash
    format: Literal["csv", "parquet", "netcdf", "geojson", "json"]
    schema: dict | None                  # optional column/field schema
    checksum: str                        # content hash; mutation invalidates DataRef identity
    schema_version: str                  # SemVer / calver for this dataclass schema
```

### 4.4 Warning Schema

Warnings are structured, not free-text strings, so downstream consumers can filter and route them programmatically.

```python
@dataclass
class Warning:
    code: str                            # e.g. "domain-mismatch", "crs-mismatch"
    severity: Literal["info", "caution", "critical"]
    message: str
    affected_output: str | None          # which output key is affected
    remediation: str | None             # suggested action
    schema_version: str                  # SemVer / calver for this dataclass schema
```

## 5. Orchestrator Specification

### 5.1 Routing Logic

The orchestrator proposes workflows; users approve consequential analytical choices before execution e.g., **method selection, parameter defaults, model assumptions**. Orchestrator uses a two-stage approach:

**Stage 1 — Rules-based classification.** Parse the request for explicit method names, file formats, and domain keywords. Handles the majority of unambiguous requests (e.g., "run a Sobol sensitivity analysis on this parameter file"). Confidence threshold: ≥ 0.85.

**Stage 2 — LLM-assisted disambiguation.** For requests below the Stage 1 threshold, invoke an LLM with a structured prompt returning a routing decision as JSON. The LLM call — including model ID and prompt hash — is logged as part of provenance.

Below 0.60 confidence after Stage 2, present routing options to the user. Do not auto-route.

### 5.2 Workflow Composition

When a request maps to multiple skills, the orchestrator:

1. Resolves execution order from declared input/output dependencies
2. Identifies intermediate outputs that must be persisted for provenance
3. Presents the proposed workflow DAG to the user before execution
4. Executes sequentially unless skills are explicitly declared parallelizable

Example workflow dispatch:

```yaml
request:
  analyze:
    - model_outputs
    - inequality
    - uncertainty

workflow:
  - sensitivity-analysis
  - statistical-analysis
  - visualization-reporting
```

### 5.3 Error Handling

| Condition                     | Behavior                                                       |
| ----------------------------- | -------------------------------------------------------------- |
| `status: "failed"`            | Halt pipeline; surface error with remediation hint             |
| `status: "partial"`           | Notify user; continue with downstream steps marked provisional |
| Warning severity `"critical"` | Pause; require user confirmation before proceeding             |
| Warning severity `"caution"`  | Log and continue; include in final report                      |
| Routing confidence < 0.60     | Present routing options to user; do not auto-route             |
| Environment spec conflict     | Fail fast before any execution                                 |

### 5.4 Provenance Coordination

The orchestrator assembles the final RO-Crate bundle from individual skill `ProvenanceRecord` outputs. Each record must include:

- Software versions (package name, version, source)
- Input checksums
- Parameters used, including applied defaults
- Execution timestamp and duration
- LLM routing decisions, if any (model ID, prompt hash)

## 6. Skill Specifications

### 6.1 Sensitivity Analysis

**Purpose:** Parameter sensitivity and uncertainty quantification for computational models.

**Trigger keywords:** sensitivity, uncertainty, Sobol, Morris, Monte Carlo, parameter importance, variance decomposition

**Methods:**

| Method                   | Library       | Notes                                                 |
| ------------------------ | ------------- | ----------------------------------------------------- |
| Morris screening         | SALib         | First-pass parameter ranking; low sample cost         |
| Sobol indices            | SALib         | Variance decomposition; requires larger sample budget |
| Monte Carlo              | SALib         | General uncertainty propagation                       |
| Latin hypercube sampling | SALib         | Space-filling alternative to Monte Carlo              |
| Exploratory modeling     | ema_workbench | Deep uncertainty; scenario discovery                  |
| Surrogate modeling       | ema_workbench | Emulator-based analysis for expensive models          |

**Required inputs:** parameter bounds file (JSON or CSV), model callable or output ensemble

**Outputs:** parameter rankings, first/total-order indices with confidence intervals, convergence diagnostics, ensemble statistics

**Human confirmation required:** when sample size N is auto-selected — surface proposed N to user before generating samples.

**Warnings:**

- `insufficient-samples` (caution) — N below recommended minimum for chosen method
- `non-convergence` (critical) — bootstrap confidence intervals on indices exceed 20% of index value

**Deferred:** OpenMOLE integration (JVM dependency; target v0.3)

---

### 6.2 Statistical Analysis

**Purpose:** General statistical description, inference, and diagnostics.

**Submodules and methods:**

| Submodule          | Methods                                                                     |
| ------------------ | --------------------------------------------------------------------------- |
| Descriptive        | Summary statistics, correlation, covariance, distribution fitting           |
| Inequality metrics | Gini, Theil, Atkinson, Lorenz curve                                         |
| Inference          | Linear/GLM regression, mixed models, Bayesian workflows, hypothesis testing |
| Diagnostics        | Residual analysis, assumption checks, bootstrap, cross-validation           |

**Libraries:** pandas, scipy, statsmodels, pymc, numpy, pysal

**Outputs:** results table (machine-readable), diagnostic figures, methods summary, uncertainty bounds where applicable

**Warnings:**

- `assumption-violation` (caution or critical depending on severity) — normality, homoscedasticity, independence
- `small-sample` (caution) — N below method-specific threshold
- `convergence-failure` (critical) — Bayesian sampler R-hat > 1.01

---

### 6.3 Geospatial Analysis

**Purpose:** Spatial and geographic data analysis.

**Pre-flight validations (run before any analysis):**

1. CRS check — all inputs must share a common CRS or be reprojected with user confirmation
2. Topology validation — invalid geometries flagged and optionally auto-repaired
3. Extent sanity check — warn if input extents are implausible for the stated region

**Methods:** reprojection, spatial join, raster analysis, spatial autocorrelation, interpolation, accessibility analysis

**Libraries:** geopandas, rasterio, shapely, pyogrio, xarray, pysal

**Outputs:** GeoJSON or GeoParquet results, figures (static + interactive), CRS and validation report

**Warnings:**

- `crs-mismatch` (critical) — inputs in different CRS, reprojection required
- `geometry-repair` (caution) — geometries auto-repaired; list affected feature IDs
- `projection-distortion` (caution) — chosen CRS introduces >1% area distortion for analysis extent

---

### 6.4 Text Analysis

**Purpose:** NLP and corpus analysis workflows.

**Submodules and methods:**

| Submodule | Methods                                                                                   |
| --------- | ----------------------------------------------------------------------------------------- |
| Sentiment | VADER, transformer classification, lexicon analysis, temporal sentiment                   |
| Discourse | Topic modeling (LDA, BERTopic), semantic similarity, embedding analysis, stance detection |
| Corpus    | Tokenization, n-gram analysis, frequency analysis, metadata linking                       |

**Libraries:** transformers, spaCy, sentence-transformers, vaderSentiment, NLTK

**Warnings:**

- `domain-mismatch` (caution) — model trained on different domain than input corpus
- `cultural-bias` (caution) — model known to underperform on non-Western cultural content
- `multilingual-limitation` (caution) — corpus contains languages outside model training distribution
- `sarcasm-limitation` (info) — sentiment results may be unreliable for ironic or sarcastic content

All warnings include `affected_output` and `remediation` fields per §4.4.

---

### 6.5 Network Analysis

**Purpose:** Graph and relational structure analysis.

**Methods:** degree/betweenness/eigenvector centrality, community detection (Louvain, SBM), diffusion analysis, temporal networks, multiplex analysis

**Libraries:** NetworkX, igraph; graph-tool as optional heavy dependency

**Outputs:** node/edge attribute tables, community assignment file, centrality rankings, summary figures

**Warnings:**

- `giant-component` (info) — graph disconnected; analysis run on largest component only
- `self-loops` (caution) — self-loops present; note impact on chosen centrality measure

---

### 6.6 Visualization and Reporting

**Purpose:** Standardized, reproducible figure and report generation. Invoked by the orchestrator as a terminal step; does not run in isolation.

**Outputs:**

| Type                  | Format                     | When                                        |
| --------------------- | -------------------------- | ------------------------------------------- |
| Publication figures   | PNG/SVG at 300 dpi         | Any workflow producing quantitative results |
| Interactive dashboard | Plotly/Altair HTML         | Explicitly requested, or geospatial output  |
| Markdown report       | `.md`                      | Default for all workflows                   |
| Quarto report         | `.qmd` + rendered PDF/HTML | Explicitly requested                        |
| Provenance bundle     | RO-Crate `.zip`            | Always                                      |

**Libraries:** matplotlib, Plotly, Altair, Quarto

**Constraint:** figures must be generated from persisted data files, not in-memory objects, so they can be reproduced independently.

## 8. Execution Model

### 8.1 Runtime

**Preferred:** Python ≥ 3.11, Jupyter or Quarto for notebooks, Docker or Apptainer for containerized execution.

**Environment management:** conda (default), uv, or pip-tools. The resolved environment spec is an output artifact for every run.

**Optional runtimes:** R (statistical extensions), Julia (performance-critical numerical work), Nextflow or Snakemake (HPC workflow orchestration).

### 8.2 State and Session Management

The orchestrator is stateless between sessions. Workflow state is persisted to a job directory and can be resumed from any completed step using the provenance record. No in-memory state should be assumed to survive a process restart.

### 8.3 Partial Failure and Resume

Completed steps are not re-run on resume. The orchestrator reads completed step outputs from the job directory and continues from the first incomplete step.

## 9. Provenance and FAIR Compliance

Every workflow execution produces a provenance bundle:

- `ro-crate-metadata.json` (RO-Crate 1.1)
- `CITATION.cff` for the analysis code
- `codemeta.json` for software dependencies
- `environment.yml` or `requirements.txt` (exact pinned versions)
- `workflow.json` — skill DAG, parameters, and checksums
- All output artifacts with sha256 checksums

This bundle is the unit of sharing and archival. Output distributed without its bundle is considered incomplete.

## 10. Acceptance Criteria

### 10.1 Per-Skill

A skill is ready for integration when it:

- Passes unit tests with ≥ 90% coverage on happy-path and documented failure cases
- Returns a valid `SkillResult` (with provenance and diagnostics) for all test inputs
- Produces identical outputs on two independent runs from the same provenance record
- Emits correct structured warnings for all documented warning codes
- Completes within the documented time budget for reference input sizes

### 10.2 Orchestrator

- Routes ≥ 95% of benchmark requests correctly at Stage 1
- Logs all LLM routing decisions with model ID and prompt hash
- Presents workflow plan to user before execution for all multi-skill workflows
- Assembles a valid RO-Crate bundle for every completed workflow

### 10.3 System

- End-to-end workflow is reproducible from its provenance record alone
- All `"critical"` warnings halt execution pending user confirmation
- No skill calls another skill directly (enforced by integration test)
